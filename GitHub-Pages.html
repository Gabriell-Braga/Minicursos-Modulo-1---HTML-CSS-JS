MÓDULO 4:
GITHUB PAGES

GIT, REPOSITÓRIO E SITE NO AR EM github.io

Do projeto local à URL pública

Você já tem HTML, CSS e JS — agora vamos versionar e publicar.

GIT E GITHUB

Git é o controle de versão na sua máquina: histórico de alterações, branches e commits.

GitHub é um serviço na nuvem que hospeda repositórios Git — backup, colaboração e (com Pages) hospedagem gratuita de site estático.

LOCAL

Pasta do projeto + pasta oculta .git com o histórico.

O QUE VOCÊ PRECISA

  • Conta gratuita em github.com.
  • Git instalado no computador (git --version no terminal).
  • Terminal integrado ao editor (VS Code / Cursor: Ctrl+`) ou PowerShell / bash.

Windows: instale o Git for Windows. macOS: xcode-select --install ou Git oficial.

Revisão rápida de Git

Comandos que você usará o tempo todo.

CONFIGURAR IDENTIDADE (1×)

Antes do primeiro commit, diga ao Git quem você é (aparece no histórico).

Terminal
git config --global user.name "Seu Nome"
git config --global user.email "seu@email.com"

Use o mesmo e-mail da conta GitHub (ou o e-mail privado que o GitHub gera em Settings → Emails) para os commits aparecerem no seu perfil.

INIT E STATUS

Dentro da pasta do projeto:

git init
git status
  • init cria o repositório local (a pasta .git).
  • status mostra arquivos untracked, modified ou prontos para commit.

ADD E COMMIT

Commit = snapshot com mensagem. Primeiro os arquivos entram na staging area com add.

git add .
# ou arquivo específico: git add index.html

git commit -m "Primeira versão do site"

Boas mensagens descrevem o que mudou, no imperativo: “Adiciona CSS do perfil”, “Corrige link do menu”.

LOG E DIFF

git log --oneline
git diff
git diff --staged
  • log — histórico de commits (hash + mensagem).
  • diff — alterações ainda não adicionadas ao stage.
  • diff --staged — o que vai entrar no próximo commit.

BRANCH MAIN

Repositórios novos costumam usar a branch principal main. Se o Git criar master, você pode renomear:

git branch -M main

git branch lista branches; git switch nome troca de branch (fluxos maiores — aqui só precisamos da main).

REMOTE, PUSH E PULL

Remote = cópia no GitHub. Depois de criar o repositório vazio no site, conecte e envie:

git remote add origin https://github.com/USUARIO/repositorio.git
git push -u origin main

pull traz alterações do remoto: git pull origin main — útil em equipe ou entre dois computadores.

CLONE E .GITIGNORE

Clone baixa um repositório existente:

git clone https://github.com/USUARIO/repo.git

Arquivo .gitignore lista o que o Git não deve versionar:

.DS_Store
Thumbs.db
*.log
node_modules/

GitHub Pages

Hospedar HTML/CSS/JS direto do repositório.

CRIAR O REPOSITÓRIO

  • No GitHub: New repository → nome (ex.: meu-portfolio) → visibilidade Public (Pages gratuito em público; contas pagas podem usar privado com Pages).
  • Pode marcar Add a README ou deixar vazio se já vai dar push da pasta local.
  • Copie a URL HTTPS do repositório (botão verde Code).

PRIMEIRO ENVIO

Fluxo típico: projeto já na pasta → commits locais → ligar ao GitHub → push.

cd pasta-do-seu-site
git init
git add .
git commit -m "Site inicial"
git branch -M main
git remote add origin https://github.com/USUARIO/repo.git
git push -u origin main

Na primeira vez, o Git pode pedir login: use Personal Access Token (GitHub → Settings → Developer settings) no lugar da senha, ou faça login pelo Git Credential Manager.

ATIVAR O PAGES

  1. Abra o repositório no GitHub → Settings.
  2. Menu lateral → Pages (em “Code and automation”).
  3. Source: Deploy from a branch.
  4. Branch: main e pasta / (root) — ou /docs se seu index.html estiver dentro de uma pasta docs.

Salve. Em até alguns minutos o site fica disponível na URL pública.

QUAL É A URL?

Projeto em repositório normal:

https://USUARIO.github.io/nome-do-repo/

Se o repositório se chama USUARIO.github.io (nome especial), o site fica em https://USUARIO.github.io/ — útil para domínio curto.

A página inicial deve ser index.html na pasta escolhida (raiz ou docs).

ATUALIZAR O SITE

Sempre que alterar arquivos localmente:

git add .
git commit -m "Descreve a mudança"
git push

O GitHub Pages regenera o site em cerca de 1 a 3 minutos. Atualize o navegador com cache limpo (Ctrl+F5) se não vir a mudança.

PROBLEMAS COMUNS

  • 404 — caminho errado; em subpastas, links relativos devem bater com a estrutura publicada.
  • Maiúsculas — no GitHub (Linux) Foto.jpg ≠ foto.jpg. Use sempre o mesmo nome que no disco.
  • HTTPS — misturar recursos http:// em página https:// pode ser bloqueado (conteúdo misto).
  • Branch — Pages apontando para branch ou pasta errada; confira em Settings → Pages.

CHECKLIST DE PUBLICAÇÃO

  • index.html na raiz (ou em docs/ se for essa opção).
  • Caminhos para CSS/JS/imagens relativos ao HTML (ex.: styles.css, ./exemplos/...).
  • Repositório Public (ou plano com Pages em repo privado).
  • Pages ligado à branch main e pasta correta.
  • Testar a URL .github.io no celular ou aba anônima.

PARABÉNS!

Você uniu HTML, CSS e JavaScript num projeto real — e agora sabe colocá-lo no ar com GitHub Pages.

Seu trabalho deixa de ser só uma pasta no PC: vira um link que mostra o que você fez. Cada push atualiza a vitrine; é um passo concreto na sua jornada como dev.

Guia oficial do Pages: docs.github.com/pages
